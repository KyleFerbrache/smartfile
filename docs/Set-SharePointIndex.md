# Set-SharePointIndex

### Purpose

`Set-SharePointIndex` is a PowerShell utility within the **SmartFile** module that normalizes the internal **Title** metadata of Excel workbooks (`.xlsx`) so that the value stored inside the document matches the workbook's base file name.

The utility exists to support **SharePoint document library indexing and retrieval workflows** where downstream queries rely on the SharePoint **Title** column rather than, or in addition to, the file name shown in the library UI.

In practical terms, the command is used to reduce metadata drift between:

- the file name as stored on disk,
- the OpenXML Title value embedded inside the workbook,
- and the SharePoint Title column that can be filtered, indexed, and queried through SharePoint APIs.


#### When to Use It

Use `Set-SharePointIndex` when all of the following are true:

1. A SharePoint library stores a large number of Excel workbooks.
2. Those workbooks are synchronized locally or otherwise available on disk for processing.
3. A downstream SharePoint integration, automation, or application uses the **Title** column in queries.
4. The file names are the authoritative identifiers for the workbook set.

This is especially useful when files are exported or generated by upstream systems that either:

- do not write Title metadata at all, or
- populate a generic or stale Title value such as **Table Export**, **Workbook**, or another non-unique default.

---

## SharePoint-Specific Operating Requirements

`Set-SharePointIndex` modifies local `.xlsx` files only. It does **not** call the SharePoint API directly and it does **not** create or manage SharePoint columns.

For the command to provide meaningful downstream value in SharePoint, the target document library should be configured appropriately.

#### Required SharePoint Condition

The library should contain a **Title** column that is **indexed**. This is a built-in column for all standard Document Libraries.

This matters because the main operational value of this utility is realized when downstream systems can filter or query efficiently against that column in large libraries. If the Title column is not indexed, SharePoint may still store the value, but API queries and filtered views are less likely to perform well at scale. In libraries of 5,000 files or more, queries may be blocked entirely if a restrictive set of **indexable** columns are not provided for pre-filtering of data retrieval as non-indexed OData filters are processed after the data retrieval.

#### Recommended SharePoint Practice

Create a **dedicated library view** for the specific fileset or workflow that will be consumed by downstream applications.

That view should apply the intended filtering logic in advance, using indexed columns wherever possible. The result is that integrations can query the view-facing dataset or reproduce the same filter pattern consistently, instead of repeatedly issuing broader library queries and filtering client-side.

#### Best Fit Scenario

This utility works best when used to normalize **one coherent fileset inside a larger library**. In large directories or large libraries, a narrow, predictable fileset is substantially easier to index, filter, and retrieve through SharePoint OData queries than a mixed-content library with weak metadata discipline.


#### Why Title Normalization Helps SharePoint

SharePoint supports OData query operations such as `$select`, `$filter`, and `$orderby` for REST requests. In large libraries, performance depends heavily on filtering strategy and indexed column usage.

If a workflow depends on a metadata field such as **Title**, the field is only operationally useful when:

- the value is populated consistently,
- the value is semantically meaningful,
- and the column is indexed and used in a threshold-friendly way.

`Set-SharePointIndex` addresses the first two concerns by making the workbook's internal Title deterministic and aligned to the file name. In environments where SharePoint surfaces and indexes the Title column for those documents, this creates a cleaner retrieval path for downstream OData consumers.

## Functional Behavior

At execution time, `Set-SharePointIndex` performs the following high-level operations:

1. Recursively scans the provided root directory.
2. Identifies `.xlsx` files within scope.
3. Optionally restricts the scan to files modified in the last **X** days.
4. Opens each workbook as an OpenXML package.
5. Checks for `docProps/core.xml`.
6. If `core.xml` does not exist, creates it with a valid OpenXML core-properties structure.
7. If `core.xml` exists, reads the current `dc:title` value.
8. Compares that value to the file's base name.
9. If the value is missing or mismatched, writes the file name into the Title field.
10. Leaves already compliant files unchanged.

The command intentionally avoids Excel COM automation and shell-property manipulation. Instead, it uses direct OpenXML editing for greater determinism, better performance, and fewer external runtime dependencies.


### File Format Assumptions

`Set-SharePointIndex` is designed for **Excel Open XML workbooks** (`.xlsx`).

This matters because `.xlsx` files are ZIP containers with a well-defined internal structure. The document Title is stored in:

```text
docProps/core.xml
```

The utility treats the workbook as a ZIP archive and updates the OpenXML core properties directly.

##### Not Supported

- `.csv`
- legacy `.xls`
- non-Excel file types
- files currently open and locked by another process


### Parameters

The exact parameter set may evolve, but the documented core execution model assumes the following inputs.

##### `-RootPath`

Specifies the local root directory to scan recursively.

This should point to the synchronized or local file path representing the SharePoint-backed fileset you want to normalize.

##### `-Days`

Optional integer value that limits processing to files modified in the last **N** days.

If omitted, all matching `.xlsx` files under the root path are considered in scope.

This parameter is intended to support incremental processing patterns where only recently changed files should be normalized.

---

### Typical Execution Pattern

A typical use case looks like this:

1. A SharePoint document library contains a large number of Excel files.
2. A local sync or operational export makes the library content available on disk.
3. `Set-SharePointIndex` is run against a known subfolder or fileset.
4. The updated files sync back to SharePoint.
5. SharePoint indexes the Title column.
6. OData consumers query against the indexed Title field or against a prepared library view built on that field.

This pattern is particularly effective when the fileset is stable, naming conventions are controlled, and downstream consumers should not need to infer meaning from inconsistent metadata.

## Detailed Processing Logic

#### Step 1: Recursive Discovery

The command walks the specified root path recursively and evaluates candidate `.xlsx` files.

If the optional date filter is present, last-modified timestamps are used to reduce the working set before any file package is opened.

#### Step 2: OpenXML Package Access

Each workbook is opened as a ZIP archive using .NET compression APIs.

This is intentionally lower-level than Excel automation, but it is faster and more predictable for metadata-only work because the workbook application model itself is never launched.

#### Step 3: Core Properties Inspection

The utility searches the OpenXML package for `docProps/core.xml`.

This file contains document-level properties including the Dublin Core Title field (`dc:title`).

#### Step 4: Missing Metadata Container Handling

If `core.xml` is not present, the command creates a new core-properties document and writes a valid Title based on the workbook file name.

This is an important behavior because many system-generated workbooks are structurally valid `.xlsx` files but were never assigned document metadata by the generating process.

#### Step 5: Existing Title Evaluation

If `core.xml` exists, the command reads the current Title. It then compares that Title to the workbook base name.

- If the Title is identical, the file is skipped.
- If the Title is missing, a new Title node is inserted.
- If the Title differs, the Title is overwritten with the base name.

#### Step 6: Summary Output

The command is designed to keep console output minimal while still exposing progress during scan and processing phases. A compact summary is emitted at the end of execution so operators can validate the scope and result of the run.

---

### Why This Utility Uses OpenXML Instead of Excel COM

There are several reasons the implementation uses direct package editing rather than launching Excel.

**Performance**

Metadata-only operations do not require workbook calculation, rendering, or object-model interaction. Excel COM adds substantial overhead when executed across many files.

**Determinism**

Direct XML updates reduce dependence on workstation-specific Excel behavior, profile state, UI automation issues, and Office installation differences.

**Reduced Operational Surface**

Avoiding Excel COM makes the command easier to run in controlled automation contexts where Office desktop applications may not be installed, allowed, or desirable.

## Operational Risks and Considerations

### Limitations

`Set-SharePointIndex` is intentionally narrow in scope.

**It Does Not**
- create SharePoint columns,
- index SharePoint columns,
- publish or modify SharePoint views,
- update list item properties through the SharePoint REST API,
- rename local files,
- or correct broader metadata design problems in the library.

**It Assumes**

- file names are the correct source of truth,
- the target workbooks are `.xlsx` files,
- the files are not locked during execution,
- and SharePoint synchronization or upload behavior will carry the updated metadata into the library lifecycle.


### Locked Files

If a workbook is open in Excel or another process has a write lock, the package update may fail.

### Sync Timing

If the root path maps to a synchronized SharePoint or OneDrive location, file sync timing may affect when the updated metadata becomes visible in SharePoint.

### Metadata Propagation Expectations

Updating the internal workbook Title improves metadata consistency at the document level, but any SharePoint-side usefulness depends on how the library, content types, crawled properties, and operational processes are configured.

### Large Directory Strategy

In very large root paths, use `Set-SharePointIndex` against the most relevant fileset rather than indiscriminately normalizing unrelated content. This aligns the utility with its intended value: improving a known retrieval path, not performing generic bulk edits without context.

## Example Usage Patterns

### Full Fileset Normalization

```powershell
Set-SharePointIndex -RootPath "C:\Data\Libraries\Finance"
```

### Incremental Normalization for Recently Changed Files

```powershell
Set-SharePointIndex -RootPath "C:\Data\Libraries\Finance" -Days 7
```

### Validation Checklist

After running the command, validate the following:

- The expected number of files were discovered.
- The expected number of files were updated or initialized.
- A sample of workbooks opens successfully.
- The internal Title matches the file base name.
- SharePoint synchronization completes successfully if applicable.
- The target library view surfaces the intended fileset.
- OData retrieval behaves consistently against the indexed Title field or the prepared view pattern.

## Troubleshooting

#### Symptom: Many Files Report Missing `core.xml`

This usually indicates the workbooks were generated by an upstream export or automation path that did not write OpenXML core document properties.

This is expected and is one of the main scenarios the utility was designed to handle.

#### Symptom: SharePoint Queries Are Still Slow

Check the SharePoint side first:

- Is the Title column indexed?
- Is the query using indexed filters?
- Is the working set too broad?
- Is the integration querying the full library instead of a narrower logical dataset?
- Does the library view return a threshold-friendly subset?

#### Symptom: Files Fail to Update

Common causes include:

- the file is open,
- insufficient file-system permissions,
- a corrupt or non-standard workbook package,
- or a sync engine temporarily holding the file.

---


## Summary

`Set-SharePointIndex` is a targeted metadata-normalization utility for Excel workbooks used in SharePoint-centric environments.

Its job is narrow but operationally valuable:

- make workbook Title values deterministic,
- align them to authoritative file names,
- and improve the quality of SharePoint-side filtering and retrieval when Title is part of the query strategy.

When paired with an indexed Title column and a carefully designed SharePoint view, the utility becomes a practical foundation for more efficient downstream OData-based access patterns.
